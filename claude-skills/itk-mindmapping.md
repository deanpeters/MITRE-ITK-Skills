# ITK: Mindmapping

Help a product practitioner apply **Mindmapping** from the MITRE Innovation Toolkit.

**Phase:** GENERATE · **Difficulty:** Beginner · **Group:** 2+ people · **Time:** 30+ minutes

> Generate insights into relationships between concepts in a flexible and engaging way to visualize information.

---

## When This Tool Fits Best

- Early discovery when the problem space is ambiguous and you need to map dependencies before scoping a PRD
- Decomposing a vague feature request into themes, sub-features, and edge cases for backlog refinement
- Synthesizing user interview findings into clustered insights that reveal Jobs-to-be-Done patterns
- Kicking off OKR planning by visualizing how candidate initiatives connect to strategic objectives
- Aligning cross-functional teams during a kickoff by externalizing assumptions about a domain

---

## Key Concepts

**Radial Hierarchy** — Ideas branch outward from a central node, with proximity to center signaling importance. This structure forces explicit parent-child relationships that surface how sub-features ladder up to a core theme.

**Associative Linking** — Drawing cross-connections between branches that aren't directly hierarchical. For PMs this exposes hidden dependencies and shared components across otherwise separate epics.

**Divergent Generation** — A non-linear ideation mode that prioritizes breadth before structure. It mirrors discovery-phase thinking where you want to surface the full solution space before converging on scope.

**Central Node** — The single anchoring topic the entire map radiates from, typically a problem statement, user goal, or feature theme. A poorly framed central node distorts every downstream branch.

**Depth Levels** — The successive layers of branches from general to specific. Shallow maps stop at obvious categories; deeper levels are where non-obvious edge cases and requirements emerge.

**Visual Encoding** — Using color, imagery, and emphasis to encode meaning like priority, ownership, or status. This turns the map into a shared artifact teams can scan and reason about quickly.

---

## Common Pitfalls to Watch For

- Choosing a central node that is a solution rather than a problem or goal, which biases every branch toward a predetermined answer and kills genuine discovery.
- Stopping at the first or second level of branches, producing only obvious categories and missing the deeper edge cases where real requirements hide.
- Treating the map as a deliverable instead of a thinking tool, so the team polishes aesthetics while never converting branches into prioritized backlog items.
- Skipping associative cross-links between branches, which hides shared dependencies and leads to duplicated work or integration surprises during delivery.
- Letting one dominant voice drive the map in a group setting, collapsing divergent generation and reproducing existing groupthink rather than surfacing new ideas.
- Never translating the finished map into structured artifacts like user stories or OKRs, leaving valuable insights stranded in a format no roadmap tool ingests.

---

## How to Help

- **Read context first.** Before asking anything, check what the user has already shared — their role, product, team size, timeline, what phase of work they're in.
- **Experienced users can skip straight to the work.** If they name this tool directly or describe a clear situation, help them run it — don't force a guided flow.
- **Use the Adaptive Decision Ladder when the situation is unclear.** Offer numbered options; the user can reply `2`, `1 & 3`, or describe freely. Skip steps that context already resolves.
- **Walk through the facilitation steps** from the full skill reference, adapting language and examples to their specific product context.
- **Surface pitfalls proactively** — especially those most likely given what they've shared.
- **Suggest complementary tools** when the conversation reveals adjacent needs.
- **Stay grounded in product artifacts**: PRDs, OKRs, roadmaps, user stories, sprint ceremonies, stakeholder briefings.
